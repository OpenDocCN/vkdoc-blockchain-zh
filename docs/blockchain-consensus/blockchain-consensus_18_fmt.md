# 区块链共识协议

- `t`：待证明目标检查点的哈希值
- `h(s)`：源检查点的高度
- `h(t)`：目标检查点的高度
- `S`：发送方对整个投票消息的签名

验证者创建投票消息，对其进行签名，并广播到网络。检查点的哈希值用于标识对应的检查点。只有当`s`在检查点树中是`t`的祖先节点，且验证者的公钥存在于验证者集合中时，该投票才是有效的。当超过三分之二的验证者对从源到目标的一条链进行投票时，该链或链接就成为了**绝对多数链接**。例如，以`cp'`为源，`cp`为目标，则`cp' → cp`是绝对多数链接。如果一个检查点是创世区块，或者它是一条已证明的检查点所在的绝对多数链接中的一个检查点，那么该检查点就是**已证明的**(justified)。准确地说，我们可以表述为：如果存在一条绝对多数链接`cp' → cp`，且`cp'`是已证明的，那么检查点`cp`就是已证明的。如果一个检查点`cp`是已证明的，并且存在一条绝对多数链接`cp → cp'`，其中`cp'`是检查点`cp`的直接子节点，那么该检查点就被认为是**最终确认的**(final)。

已证明的检查点不被视为最终确认，因为可能存在冲突的已证明检查点。为了最终确认一个检查点`cp`，需要进行第二轮的确认，即`cp`的直接子节点`cp'`通过绝对多数链接`cp → cp'`被证明。

## 协议步骤

- 代币持有者在网络中存入一笔质押金。
- 在一个共识周期内：
    - 识别有效的检查点。
    - 将它们添加到检查点树中。
    - 广播对某个检查点对（源，目标）的投票。
    - 评估接收到的投票的有效性，根据罚没规则和投票签名者的质押金进行检查。
    - 如果该检查点对获得了超过总质押金三分之二的支持，则该检查点被视为已验证。
    - 该检查点对被证明并最终确认。
    - 重复此过程。

本质上，该过程分三步进行。首先，对某个检查点进行投票；如果获得了超过 2/3 的投票，则该检查点进入已证明状态。最后，链被最终确认以形成规范链。证明并不等同于最终确认；在区块树中可能存在多个已证明的链。当两个连续的检查点都获得 2/3 的投票时，就达成了最终确认。

## 安全性与活性

任何违反这些条件的验证者都将受到罚没其质押金的处罚。这些被称为最低罚没条件：

- 验证者不得发布针对同一目标检查点高度的两个不同投票。
- 验证者不得在其另一个现有投票的源到目标范围内再次投票。

有两个安全性与活性要求：

- 可追责安全性
- 可行的活性

可追责安全性与传统共识协议中的安全性要求略有不同。安全性简单地意味着，除非超过三分之一的验证者偏离协议，否则两个冲突的检查点永远无法被最终确认。可追责性意味着行为不端的验证者可以被识别并受到处罚。

可行的活性意味着，只要存在扩展已最终确认链的子节点，就总能添加绝对多数链接以产生新的已最终确认检查点。

## 总结

本章讨论了比特币之后出现的区块链时代协议。区块链共识协议有多种类型；有些基于投票，有些是工作量证明，另一类是权益证明协议。所有这些协议都具备一定的安全性和活性属性，以确保协议在给定环境中是正确且能正常运行的。我们在本章中讨论了其中几个要点。这些协议的对比总结将在最后一章，即第 10 章中给出。

我们还详细讨论了诸如`ETHASH`、权益证明及其不同类型、以及包括`HotStuff`和`Tendermint`在内的`BFT`变体等协议。本章还介绍了现代协议，如`Casper FFG`、`Solana`的历史证明协议、以及`Polkadot`的`GRANDPA`和`BABE`协议。在本章中不可能涵盖所有协议，但至少提供了足够的信息来帮助理解区块链中使用的不同类型的算法。我们未涉及的一些其他协议，例如`PoET`、`HoneyBadger BFT`和`Snow`，将在最后一章中简要介绍。
