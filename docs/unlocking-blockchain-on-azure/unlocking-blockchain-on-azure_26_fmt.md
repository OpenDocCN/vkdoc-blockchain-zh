# 第二个挑战

一旦选定了加入机制，但在成员加入之前，解决方案架构师需要确保所有活动/交易都遵循在加入前已定义的通用共识规则。例如，只有当一组智能合约规则得到验证后，定义的活动才能发生；或者可以采用权益证明机制，由大多数利益相关者决定执行或拒绝某项活动。在公私链联盟中，区块链中可以存在多种共识形式，每种形式分别适用于私有链或公有链。

因此，设计的第二个挑战是`共识机制的选择`。

形成区块链的原始原因之一是可信度；因此，共识机制需要确保贸易参与者的可信度能够在链上被透明地跟踪和维护。可以在本书定义共识机制的相关章节中识别需考虑的要点，或者也可以开发自己的机制。

针对当前用例，我们选择了工作量证明与权益证明相结合的方式，并辅以创造可信度价值的智能合约条件，从而管理整体的贸易信任。

一旦在区块链中定义了共识，就可以开始向私有/公有区块链联盟导入成员。公有链可以托管所有进行交易或参与交易的同类节点。私有链则可以在交易启动时临时形成或同步，并且只包括那些特定的贸易参与者。
