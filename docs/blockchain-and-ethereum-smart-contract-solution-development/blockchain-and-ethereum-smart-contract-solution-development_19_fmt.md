# 第 2 章：支撑区块链的核心技术概览

分布式系统中的其他计算机是否正常运行，它们可能宕机、在线或部分功能失效。其次，分布式系统中的每台计算机都无法确知网络中其他所有计算机是否可信。它们是否按设计运行？是否已被入侵？它们发送的消息是否存在恶意或欺骗性？

克服这两个不确定性的问题被称为实现拜占庭容错[\[1\]](#index_split_001.html#p76)BFT。

大多数分布式系统通过称为共识协议的算法来实现拜占庭容错。共识协议帮助分布式系统中的计算机确保来自其他计算机通信的真实性。若分布式系统中至少 51%的计算机按预期运行，若干 BFT 算法即可确保系统准确运行。若 51%或更多的计算机恶意串通，它们可能出于邪恶目的入侵分布式系统。由于分布式系统由独立计算机组成，计算机数量越多，51%的计算机串通就越困难。例如，若分布式系统仅有三台计算机，则需两台计算机串通；但若系统包含 1000 台计算机，则需 501 台计算机串通。两人串通合谋显然比 501 人串通容易得多。

## 3. 一致性

分布式系统需要缓解的第三个现实问题直接源于应对复杂性和故障的方法：即分区与复制。当为提升可扩展性而对计算与数据进行分区，并为实现容错而复制数据与计算时，确保所有副本或分区之间的一致性至关重要，但极具挑战。

如何确保这些通过网络连接的计算机不仅在语法上，而且在语义上相互一致？

分布式系统实现一致性目标的一种方式，是就如何同步以及同步频率达成严格协议。这些协议预先设计在系统中，包含交换的消息以及能确保消息真实性的算法。`校验和`可视为此类机制的一个极简案例。通过`校验和`，每份数据被分配一个`校验和`值，该值可以是消息中比特位的总和、消息中的比特位数，或是所传输数据的哈希值。`校验和`随传输数据分开传输。接收数据的计算机通过算法计算`校验和`，若计算出的值与传输的`校验和`不匹配，则表明传输过程中存在错误。

用于实现一致性的机制会对性能（速度）产生约束，并增加分布式系统实现的成本（有时相当显著）。因此，所实现的一致性机制代表着设计权衡，这些权衡基于分布式系统所解决的业务问题进行工程化设计。随着时间的推移，针对一致性问题已研究、设计并实现了两种不同的方法。

这些首字母缩略词被称为 `ACID` 和 `BASE`。

### ACID 和 BASE 语义

在设计实现涉及关键任务型真实世界数据的系统时，若数据的精确性与准确性至关重要，则首选 `ACID` 方案。`ACID` 代表**原子性（Atomicity）**、**一致性（Consistency）**、**隔离性（Isolation）**和**持久性（Durability）**。原子性意味着，当对数据执行操作时——无论数据位于单个位置，还是跨多台计算机进行分区和/或复制——该操作要么完全成功，要么彻底失败。原子性的重要性通常通过银行转账示例来说明。当我们从一个银行账户（账户 A）向另一个账户（账户 B）转账时，此操作由多个部分组成：从账户 A 取款、更新账户 A 的可用余额、向账户 B 存款以及更新账户 B 的可用余额。如果这些部分中的任何一个失败，则整个交易失败；我们绝不能陷入资金已从账户 A 取出但未存入账户 B 的状态。一致性意味着，一旦对数据的操作成功完成——无论此操作是在单台计算机上执行，还是跨越了多台计算机且数据分布于一台或多台计算机上——便能保证所有计算机上的所有数据完全相同。隔离性指的是，当多个操作并发执行时，每个操作都是独立的，不受其他操作（或其他操作的中间状态）的影响。原子性、一致性和隔离性是相互交织的；你不可能拥有其中一个而缺少另外两个。`ACID` 的最后一个组成部分是持久性，这意味着一旦写入操作成功完成，数据将无限期地持久存在。顺便提一下，如果写入操作是删除数据，那么数据将永久消失。

分布式数据库关系管理系统在实现 `ACID` 能力（或语义）方面投入了大量精力，因为关系数据库管理系统通常支持那些需要保证数据准确性的业务流程。在集中式系统（计算和数据都位于单个物理计算机内）中实现 `ACID` 语义，要比在分布式系统中实现 `ACID` 语义简单得多。

`BASE` 语义与 `ACID` 截然不同。`BASE` 代表**基本可用（Basically Available）**、**软状态（Soft State）**和**最终一致性（Eventually Consistent）**。当系统处理的数据并非关键任务型，且所有数据无需始终保持精确时，这些语义是适用的。“基本可用”意味着系统的某些组件可能已宕机，但就用户而言，他们仍感觉系统是可用的，即使并非所有功能都始终可用。软状态意味着系统的状态可能随时间变化，并且即使没有任何用户输入，也可能发生变化。最终一致性意味着，只要有足够的时间，并且在足够长的时间内没有输入，我们就能保证所有不同的副本最终将变得一致。我们发现，在内容驱动的基于 Web 的架构中（我们并非始终处理关键任务型数据），实现 `BASE` 语义能够满足我们的业务需求，因此 `BASE` 语义已为这类系统所接受。这里的“可接受”一词是关键。虽然我们无法接受银行余额在没有我们发起特定操作的情况下发生变化，但我们可以接受网站上的内容不时更新。
