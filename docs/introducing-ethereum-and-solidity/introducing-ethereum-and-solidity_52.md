# 原型设计建议

关于 Solidity 合约的原型设计，首先要了解的是，你未必需要以太坊节点来测试合约。你可以使用以太坊虚拟机合约模拟器 [`github.com/EtherCasts/evm-sim/`](https://github.com/EtherCasts/evm-sim/)。当开发者无法接入测试网时（例如使用上网本工作），该模拟器可让他们在隔离的环境中测试合约。

以下是使用真实以太币进行测试阶段时的其他原型设计最佳实践：

-   每个合约不要存放太多以太币，并尽量为合约可持有的资金设置上限。这是防止因漏洞导致资金被困的有效安全措施。在测试阶段使用真实以太币时，千万不要存放过多资金。
-   保持合约模块化且易于理解。尽可能将功能抽象到可单独测试的库中。限制变量数量和函数长度。为所有内容编写文档。
-   采用“检查-效果-交互”模式。这意味着你不应编写需要等待其他合约返回数据才能继续的程序，否则会导致超时。通常来说，你可以通过在对状态进行更改之前对返回的数据进行检查来避免这种情况。
-   编写你自己的中介机制。由于以太坊虚拟机是一个要求严苛的平台，你有责任为自己的程序创建充当安全措施的机制。
-   正如第 5 章的令牌合约教程所述，开发者正就某些类型合约的标准达成共识。你可以将合约注册到诸如 Etherchain 之类的第三方服务上，以便其他人使用。你可以在 [`etherchain.org/contracts`](https://etherchain.org/contracts) 看到公开列出的合约。
-   测试、测试、再测试！你可以在 [`test.eth.guide`](http://test.eth.guide) 找到测试资源。

你已经看过了几个明显是为演示目的而编写的合约。在服务于真实去中心化应用时，你可以创建哪些简单的智能合约？部署它们的最佳方式是什么？这些将是本章最后一节的主题。

## 第三方部署库

部署更复杂的智能合约并将其连接到 Web，略超出于本书的范畴——部分原因在于这是一个快速发展且不断变化的领域。截至撰写本文时，这也有相当难度，并需要一些耐心。因此，开发者工具是以太坊社区积极开发的一个主要领域。

领先的开发者团队已经创建了工具，使合约和去中心化应用的部署变得更加容易。以下是一些你应该了解的项目：

-   Monax 教程与 Solidity 合约
-   OpenZeppelin 智能合约
-   Truffle 部署、测试与资产创建环境
-   Dapple，面向复杂合约系统的开发者环境
-   Populus，用 Python 编写的合约开发框架
-   Embark，用 JavaScript 编写的去中心化应用开发框架
-   Ether Pudding，一个包构建器
-   Solium，一个 Solidity 代码检查工具

本书无法涵盖所有去中心化应用指南、教程、最佳实践和示例项目。你可以在 [`dapp.eth.guide`](http://dapp.eth.guide) 找到所有这些工具和库的最新链接，以及更多资源。

此外，你可以在 [`help.eth.guide`](http://help.eth.guide) 找到一系列 Gitter 频道，在那里你可以获得开发和部署方面的帮助。
