# DApp 合约数据模型

要部署一个可运行的合约，你首先需要了解的是：可以在 EVM 中存储哪些类型的数据，以及将它们存储在何处。

正如我们在前几章中讨论的那样，以太坊网络中的每个合约地址都为其智能合约提供了存储空间。这个存储空间没有限制，唯一限制是你愿意支付多少费用。截至本文撰写时，每千字节的存储空间成本约为 0.018 美元。

`Solidity` 语言使得将合约用作小型关系型数据库变得十分简单。为了进一步简化这一过程，`Solidity` 语言提供了两种我们尚未提及的常见数据类型：
- 映射（Mappings）
- 结构体（Structs）

要了解如何在 `Solidity` 中使用这些类型，请查阅 [`solidity.eth.guide`](http://solidity.eth.guide)。

在最基本的层面上，合约的专属存储空间是一个键/值存储，拥有 2²⁵⁶ 个可能的键以及相同数量的值。这个空间足以容纳你所能想象的几乎任何类型的数据库结构。

> **注意：** 回忆一下，开发者有时会将对象属性称为*键*，就像在短语*键/值对*或*键/值存储*中那样。在我们的日常例子中，一个键/值对可能是 `footSize = 11`。一个存储了专用服务器上所有人脚码大小的表格，就是键/值存储的一个例子。作为一台有状态的交易机器，你可以把整个 EVM 想象成一个巨大的键/值存储，用于记录账户余额。

希望到现在为止，你已经能构想出可以在 `Solidity` 合约中创建和使用的简单数据结构了。在下一节中，我们将开始剖析分布式应用的架构。

## EVM 后端如何与 JS 前端通信

以太坊网络与我们称之为 *HTTP 网络*（也就是万维网）之间的鸿沟，确实是能够跨越的。假设一位客户在基于 dapp 的网站上，通过传统网络浏览器输入了一个午餐订单。为了成功地将她的订单数据（例如，几杯奶昔？）在浏览器和 EVM 之间传递，dapp 的前端必须以特定格式将数据“发送”到 EVM。

> **注意：** Dapp 可能不需要拥有自己的一套合约；相反，它们可以调用其他合约中的某些公开函数来利用其功能。对于智能合约中声明为 `public` 的每个函数，`Solidity` 会自动创建一个访问器函数，以便其他合约可以调用它。

在计算领域，*数据交换格式* 的工作原理类似于国际邮政服务。虽然世界各地的不同服务器可能运行着不同的操作系统，使用不同的语言编写，由完全不同的人开发，但它们最终必须在某个时刻与一个与自己不同的服务器交换数据。

为了确保“翻译”正确，程序员会将其程序设计成以特定的*符号表示法*将信息发送给其他程序。通常，这种符号表示法描述的是整个对象的格式（在第一章中定义为属性和值的集合）。例如，一个人类数据对象可能包含身高、体重、眼睛颜色、脚码大小等。

### JSON-RPC

在当今的 Web 应用中，JavaScript 代码可以通过一种常见的对象符号表示法，即 *JavaScript 对象表示法*（*JSON*），来在 Web 上传递信息。JSON 对象可以包含数字、字符串以及某些属性的有序值序列。

在 `Web3.js` 中有两个重要的数据对象，它们在以太坊驱动的应用前后端之间的传递方式与 JSON 大致相同。它们被称为 *JSON-RPC 对象*，随 `Web3.js` 库一起提供。下面会介绍 `Web3.js` 的安装。这两个对象的使用方式如下：
-   `web3.eth` 专门用于与区块链的交互。
-   `web3.shh` 专门用于与 Whisper 的交互。

*Whisper* 是一个私有消息协议，它本身是更大的以太坊协议的一部分。你将在第 11 章中了解更多关于 Whisper 及其在路线图中位置的信息。

在运行过程中，你可以将 JSON-RPC 对象想象成在前端（HTTP 网络）和后端（以太坊网络）之间不断来回传递。

## Web 3 时代即将到来（几乎）

名为 `Web3.js` 的 JavaScript 库是新的 *Web 3 规范* 的一部分。你可以在 [`github.com/ethereum/web3.js/`](https://github.com/ethereum/web3.js/) 找到 Web 3 项目的 GitHub 页面。

*Web 3* 是去中心化网络的通用术语，就像 Web 2 由基于 Web 的应用程序和服务所定义一样。Web 1 指的是最初的万维网，它托管静态页面。自那以后，超文本传输协议一直在不断演进，以增加更多方法，并支持日益复杂的内容和脚本。

Web 3 在很大程度上是一个以以太坊协议为核心的愿景。它通常被认为包含三个组成部分：
-   点对点身份和消息系统
-   共享状态（区块链）
-   去中心化文件存储

前两个部分已经完成：以太坊网络已经上线并运行，交易正常进行！第三个部分，即去中心化文件存储，是 Swarm 项目的一部分，你将在第 11 章中了解更多相关信息。

在 Web 3 范式中，没有 Web 服务器。没有缓存、反向代理、负载均衡器、内容分发网络（CDN）或传统大规模 Web 应用部署的其他遗留痕迹。即使是去中心化的域名服务器（DNS）也将是免费的。当 Swarm 存储上线时，它会像以太坊的 Web 托管组件一样廉价。

对于所有类型的开发者和黑客来说，Web 3 颠覆了“免费增值”的应用部署模式——在这种模式下，越来越多的用户和规模会带来越来越高的托管费用。在 EVM 中，你可以通过编写高效的代码来控制成本，并且你可以确信，从第一天起，地球上的任何人都能访问你的应用。

让我们把视线拉回到 dapp 开发的具体细节，看看我们今天所熟知的 Web 是如何与 EVM 通信的。

## 使用 JavaScript API 进行实验

在第 6 章中，你已经看到通过在 `Geth` 的 JavaScript 控制台中输入命令来与 EVM 交互是多么容易。当你这样做时，实际上只是在调用以太坊 JavaScript API 附带的各个 JavaScript 方法。你输入到 `Geth` 控制台中的这些 JavaScript 方法，是由 `Geth` 独有的、类似 JIT 的 JavaScript 解释器来解释执行的。这被称为 JSRE 的*交互式*使用，或者以*交互模式*使用它。

然而，以太坊 JavaScript API 的方法也可以暴露给普通的 Web 应用，从而使它们能够与 EVM 进行通信。

### 使用 Geth 部署去中心化应用

虽然其他以太坊客户端也很流行，但 `Geth`（由谷歌开发的 Go 语言编写）及其对 JavaScript 的简易解释，使其成为将传统 HTTP Web 上的前端 Web 应用与后端 EVM 合约连接的最快方式。

由于这些由 `Geth` 解释为 EVM 代码的 JavaScript 方法可以串联成脚本（这自然是 JavaScript 的首要用途），因此这种使用方式被称为*非交互式*使用。

#### 注意

JavaScript API 的非交互式使用正是我们称之为*计算机编程*的全部原因。总的来说，这就是编程或编写程序的目标：将原本需要手动在终端中输入的命令（比如你安装 `Geth` 时输入的那些命令）自动化。在执行复杂计算或构建分析模型时，这些指令串可能会变得冗长且令人厌烦。

通过将指令串写入纯文本文件，程序员可以使程序变得简洁、快速、高效且可重复。

编程的另一个目标是将操作员需要手动输入的任务分离出来，并让它们在*线程*中并发执行，从而缩短整个任务所需的时间。正如你在 `Geth` 首次启动时所看到的，当它在同步时，你无法对该命令行窗口进行任何操作，实际上只要 `Geth` 在运行，那个线程就不会停止。

通过在 `Geth` 之上构建一个控制台，Ethcore 的开发人员允许你（控制台的操作者）在 `Geth` 于后台（你本地机器的另一个线程中）同步的同时，发出命令。

接下来，你将了解用于连接 EVM 作为后端的理想 Web 开发框架。

## 将 Meteor 与 EVM 结合使用

如果你是一名 JavaScript 开发者，你可能听说过 `Meteor.js`，这是一个让你编写响应式 Web 应用的库，这些应用可以在服务端和客户端运行对称代码。

这个全栈框架非常适用于实时 Web 应用，它之所以对以太坊前端开发有用，是因为它非常适合编写*单页应用*（简称*SPA*）。

以下是这么多以太坊开发者喜爱 `Meteor` 的原因：
-   它及其工具完全用 JavaScript 编写。
-   开箱即用，即可获得完整的开发环境。
-   部署超级简单。
-   界面完全响应式（类似于 `Angular.js`）。
-   使用名为 `MiniMongo` 的 NoSQL 数据模型，可自动持久化到本地智能合约存储。

要了解有关使用 `Meteor.js` 构建以太坊应用的更多信息，请查看 [`github.com/ethereum/wiki/wiki/Dapp-using-Meteor`](https://github.com/ethereum/wiki/wiki/Dapp-using-Meteor)。此 URL 也列在 `tutorials.eth.guide` 上。接下来，你将学习如何将 `Web3.js` 库安装到你的开发机器上，以便开始本地使用合约。

### 安装 Web3.js 以构建支持以太坊的 Web 应用

`Web3.js` 库通过 RPC 与本地节点通信。该库可与任何以太坊节点配合使用，只要该节点暴露了其 RPC 层。你需要在本地机器上安装此库以进行开发，并在 Web 服务器上安装它以运行你的前端应用。

即使在 `Geth` 中启动链时未标记此命令，私有链默认也会暴露该 RPC 层。

实际上，你可以将以太坊节点视为裸机层，通过其 RPC 层暴露 EVM。该 RPC 层可以与同样运行 `Web3.js` 的 Web 服务器发送和接收 `web3.eth` 和 `web3.shh` 对象。

要在本地开发环境中安装 `Web3.js`，请打开终端并使用你最熟悉的安装库：
-   *npm*：`npm install web3`
-   *bower*：`bower install web3`
-   *meteor*：`meteor add ethereum:web3`
-   *vanilla*：链接 `dist./web3.min.js`

然后你需要创建一个 `Web3` 实例并将你的本地主机设置为提供者。要继续学习如何使用 `Web3.js`，请访问 [`dapps.eth.guide`](http://dapps.eth.guide)。接下来，你将看到如何在 `Geth` 控制台中执行 JavaScript 文件。

## 在控制台中执行合约

关于去中心化应用部署的完整教程会占据很多篇幅，并且可能有数十种实现方式。本节则侧重于快速入门。

你可以直接通过 `Geth` 上传你的智能合约文件，只需添加 `--exec` 参数，然后编写指向本地脚本的 JavaScript 代码，即可通过交易将其发送到 EVM。例如：

```
$ geth --exec 'loadScript("/Desktop/test.js")'
```

实际上，你甚至可以执行位于另一台机器（只要它运行了 `Geth`）上的 JavaScript：

```
$ geth --exec 'loadScript("/Desktop/test.js")' attach https://100.100.100.100:8000
```

下一节将介绍支持以太坊的应用的架构，以及它们与传统 Web 架构的不同之处。

### 合约如何暴露接口

在使用 JavaScript 去中心化应用 API 时，通过诸如 `eth.contract()` 函数这样的抽象层调用合约，将返回一个对象，该对象包含该合约可以通过 JavaScript 调用的所有函数。

为了标准化这种自省功能，以太坊协议附带了一个称为*应用二进制接口*（简称*合约 ABI*）的功能。ABI 的行为类似于 API，为应用调用合约创建了标准语法。

ABI 规定合约将返回一个数组，该数组描述了正确的调用签名和可用的合约函数。

可能会让一些开发者（尤其是来自苹果开发环境的开发者）感到惊讶的是，并没有“随”以太坊一起提供的框架来简化常见应用组件的编写。

尽管以太坊协议通常可能功能匮乏，但在常见用例中，仍然需要让合约以可预测的方式交互。这些场景包括货币单位、名称注册和交易所交易。ABI 正是对这些场景的妥协。

ABI 包含*二进制*这个词，因为在 EVM 中，应用层之下的层是运行 EVM 字节码的层。

你可以在 [`github.com/ethereum/wiki/wiki/Ethereum-Contract-ABI#functions`](https://github.com/ethereum/wiki/wiki/Ethereum-Contract-ABI#functions) 或 [`abi.eth.guide`](http://abi.eth.guide) 找到此规范。智能合约的标准通常包含一些常用方法的函数签名集合，例如发送、接收、注册、删除等。
